中间代码优化 Optimizations

6.1 什么是中间代码优化

优化对象

  • 首先，中间代码优化不是指令调度
    • 指令调度的优化是和具体的机器有关的、low - level 的层次的优化；
  • 而中间代码的优化，对象可能是抽象语法树，或者一个线性的、ILOC 的代码的优化；
    • ILOC 并不和具体的机器有关；这部分的优化是不依赖于具体的机器的；

Constant Propagation 和 Constant Folding 区别

  • 它们两者可以互相驱动

6.2 其他中间优化技术

6.2.1 优化范围

Optimizations Scopes

  • BB
    • 当进行优化时，我们面对的是 local scope，即面对的是 Basic Block；
  • Region
    • regional scope，并不是直接面对整个函数，但是函数可能被切分成好几个 BB，需要对这些 BB 构成的区域进行优化；
  • Fuction
    • 单个函数
  • Module
    • 多个函数构成的模块，但不是全部的程序；
  • Whole Program
    • 当面对 global scope 时，面对的是 whole procedure/intra-procedural；
  • 当面对 inter-procedural 时，其默认为关闭，这种优化是十分昂贵的；
    • Assume that on avg each operation takes 10 cycles；
    • machine can do 10 的八次方指令 -> 每秒；

BB 大小和程序特点

  • 小 Basic Block 程序
    • 程序当中有很多的 loop、if-else、switch 的程序，其 BB 会被切分的比较小；
  • 大 Basic Block 程序
    • 程序主要用于进行科学、数学计算，程序指令大量线性排列，其 BB 会比较大；

6.2.2 成本效益分析

cost benefit analysis
分析优化带来的收益是否大于成本；

  • 有可能优化带来的成本会大于收入；
    • 比如 gcc 的 -O 2 和 -O 3 不一定就是 O 3 的性能更好；

提取哪些元素，从 Memory 到 L1 cache 中；

  • 寻址性能

Fuction Inlining
直接把函数的代码插入到引用的地方；减少了函数的跳转、堆栈的使用；

6.3 配置文件引导的优化

Profile Guided Optimizations | PGO

什么是 Profile
Profile 是收集一些静态的关于程序的信息，比如函数、loop 执行频率、程序执行时间、每个函数花费的时间等等；

  • 比如
    • Fun （）花费了 40%的时间；
    • Gun（）花费了 23%的时间；
    • F 40（）花费了 12%的时间；
    • //…
    • 因此知道了：当前程序的 75%的执行时间，都花费在了这三个函数上；对于这三个函数之外的函数的优化，对性能帮助不大；
  • Profile -> 需要对这三个函数进行优化；
    • 一些编译器可以运行你编译程序时，可以再次编译、根据得到的 profile 文件；
  • 而这些 profile 信息，很多取决于当前程序的输入 Input；
    • Profile 依赖于 Input ；

从 Profile 到优化：内联函数
可以用于优化内联函数；
比如：当前 loop 当中有函数调用，Profile 可以估算当前 loop 的执行频率，进而帮助决定是否对函数进行内联；

从 Profile 到优化：寄存器分配
每个 BB 被执行多少次、什么时候被执行；spill 将被分配到更少频率被使用的变量上，减少 load 和 store 的次数；

从 Profile 到优化：确定优化等级
很多编译器，比如 GCC 可以选择优化等级，比如 O 1 O 2 等；

  • 因为编译器并不能知道太多程序编译后运行的具体信息，因此更高的优化不一定带来更好的性能；
  • 并且，编译器的优化是谨慎的（conservatively）；
    可以使用 Profile 来帮助确定那种优化的性能更好；
