4.1 局部寄存器分配

Local Register Allocaton

基本前提：因为是局部寄存器分配，因此假设：
The Code has been broken into basic block
每个 BB 有其局部的 ILOC；

选择一个 clear 寄存器 for Spilling，将会节省我们的存储（Store）操作；

  • 当一个寄存器在被分配到一个变量后，如果此变量会在后续代码被重复的使用，那最好就是让其一直持有此物理寄存器，减少 Store 的操作；

4.1.1 局部寄存器分配总览

什么是寄存器分配
在经过了 Code 生成阶段后，程序已经生成了基于 CFG 得到的中间代码；

  • 此中间代码独立于具体的平台，和具体的寄存器数量无关。因此寄存器分配阶段，需要根据当前寄存器的数量限制，为这些中间代码分配具体的寄存器，并且将不再使用的变量存储到内存当中；
  • Virtual Register -> Local Register Allocation -> Spilling
    • Local：基于 Basic Block 的寄存器的生命周期；
    • Spilling：当当前寄存器数量不够时，从其他的已分配变量上拿寄存器；

4.1.2 物理寄存器分配的函数实现

当得到了虚拟寄存器后，需要将其分配物理寄存器。但物理寄存器数量有限，因此就需要对有限的物理寄存器进行算法设计，然后最高效的分配；

使用例子

  • 假设 ASSUME：只有两个物理寄存器

LocalRegAllc 函数
为每个操作数的寄存器进行判断，执行物理寄存器函数分配；

Ensure 函数
确定每个虚拟寄存器当前是否有对应的物理寄存器，如果没有，就为其执行 Alloc 函数，确保被分配到物理寄存器；

Alloc 函数
为虚拟寄存器分配物理寄存器；

Free 函数

4.2 全局寄存器分配

前言回顾：局部寄存器分配的问题 -> 不允许你分配物理寄存器维持到 BB 之外；

不同的 BB 之间，无法保持寄存器的内容在不同的 BB 之间保持；

  • 局部寄存器分配的问题
    • 需要更多的 Load 和 Store 来保证变量可以获得寄存器，尤其是跨 BB 之间

4.2.1 局部寄存器分配问题

BB 加载

  • 当进行 Local 的分配时，需要：
    • 存储 Store 所有的 liveout，在 def 的 BB 结束时；
    • 需要 Load 所有的 livein，在进入 use 的 BB 时；
  • 需要大量的 Load 和 Store；

Livein
在某个 BB 中的某些变量在另一个 BB 中被定义，然后再当前 BB 中被使用；

  • 变量的 def 在当前 BB 之前；

Liveout
当前 BB 中的变量 def 后，还需要被其他 BB 使用；

  • 变量的 def 将会被在 BB 之后使用；

图示：BB 之间 Live 关系

4.2.1.1 复杂案例分析

BB 1 -> BB 2 , BB 3

  • liveout = {a}
  • livein = {}

BB 2 -> BB 4

  • livein = {a}
  • liveout = {a, d}

BB 3 -> BB 4

  • livein = {a}
  • liveout = {a, d}

BB 4

  • livein = Union ( liveout 3, liveout 2) = {a, d}

4.2.1.2 Local Live Range 分析

在线性的 BB 代码内部当中，live range 是在其内部被定义的；
如果 live range 是覆盖了超过两个虚拟寄存器，你无法为其同时保持物理寄存器（假设只有两个物理寄存器）；

  • a: 1-7
  • b: 3-5
  • c: 6-8

4.2.2.2 变量寄存器的 Spill 决策

什么是 Spill Cost
Spill Cost 是变量 spill 它们的 live range 时，需要的花费；

  • Spill a live range -> 表示一个变量 X 的花费：插入一个 Store 在每个 Def 之后，以及 Load 在任何一个 Use 之前；
  • 假设 a 需要被 Spill
    • def a 之后，需要 store a；
    • use a 之前，需要进行 load a，每个 use 的地方之前都需要；

决定一个变量是否被 Spill 的两个因素

  • 因素一：def 和 use 的数量
    • 如果一个变量 x 的 def 和 use 数量很多，那就会产生很多的 store 和 load，因此就不应该被 spill；
  • 因素二：BB 是否在 Loop 中
    • 尽管一个变量 x 可能只在一个地方被 use 或 def ，但如果这个地方在 loop 中，那也会被 store 或者 load 很多次；
    • 因此需要计算 Loop 与 spill 受影响的频率；
      • 但编译器不知道编译时，这些 Loop 将会执行多少频率；
    • 编译器的评估机制
      • 不精确的评估；
      • 分析有哪些变量是在 Loop 之外，比如 b 是在 BB 4 之外；因此 b 是第一个被 Spill 的；
      • 如果还需要 Spill，那就 Spill d，因为 a 会在比 d 更多的地方被使用到；BB 1、2、3、4 都有它，所以 Spill 的成本更高；

Live Range 和变量 redef 重定义
In for ming live range, we must have for every def , all the use that are reach by that def must be in the same range
And for every use ,all the defs that reach that use must be in the same range

4.2.2.3 两物理寄存器下的分析

基本假设

  • load 的代价是 2
  • store 的代价是 1

计算方法

  • def 的 cost 计算
    • def 就是 write，就是生产一个输出操作符；
    • number ✖ ( defs ✖ cost of store ✖ 频率)
  • use 的 cost 计算
    • use 就是 read，就是生产一个输入操作符；
    • number ✖ ( uses ✖ cost of load ✖ 频率)
  • def 和 use
    • 当 def 和 use 需要被 sipll 的时候，def 和 use 就是各自创建了一个小的 live range；
    • def -> store；两行代码是一个 live range；
    • load -> use；两行代码是一个 live range；
    • 它们有可能会导致一个原本常的 live range 被迫被分为几个比较小的 live range；

图示

分析

  • Spill cost A
    • a = 1*1 + 1*2*0.5 + 1*2*10 = 22
    • 第一个 1*1 是 a 定义的代价；
    • 第二个 1*2*0.5，其中 2 是 load 的代价，0.5 是进入 BB 3，BB 3 权重带来的代价；
    • 第三个 1*2*10，2 是 load 的代价，10 是循环 10 次的代价；
  • Spill cost for B
    • b = 1*1*0.5 + 1*2*0.5 = 1.5
  • Spill cost for C
    • c = 1.5
  • Spill cost for D
    • d = 1*1*0.5 + 1*2*0.5 + 1*1*0.5 + 1*2*10 = 21
  • Spill cost for E
    • e = 1*1*1 + 1*2*1 = 3

冲突图与代价：

  • 节点当中的 c 和 e 的物理寄存器不需要考虑是否足够的问题，因为 a 和 c、e can be colored without a problem

BB 与 Spill 分析

因为 b 和 d 的 cost 比 a 要更小，因此它们两个需要被 spill；因此被 load 和 store

  • 带来的影响
    • b 和 d 的 live range 变小；
    • A load defines a rge
    • A store uses a reg

总结

  • 在当前的路径中，变量在冲突图分析后得知：其所需的物理寄存器和当前所拥有的物理寄存器满足，则不需要 spill；
  • 当需要 Spill 时，选出当前 Spill 的 cost 中最低的变量，对其进行 Spill -> 所有 def 之后 store，所有 use 之前 load；
