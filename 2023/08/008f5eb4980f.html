    2. 网络协议

    2.1 网络协议

    • 核心：解决两台电脑之间的信息传递

    • 分层：本质：将复杂问题进行分层抽象，忽略其底层实现细节

    • OSI模型：网络七层模型

    • 网络游戏编程通常只需要Socket-Based

    • TCP协议

    • 协议特点

    • 特点一：通过三次握手保证信息的正确性

    • 特点二：保证数据的顺序；前面的包和后背的包顺序保持

    • 特点三：可以进行流量控制|网关对TCP友好

    • 特点四：TCP的Header文件大小大

    • 确认接受

    • 发送ACK内容

    • 拥塞控制

    • 不稳定的公网出现时，网络会出现波动，但也保证了网络不崩溃

    • UDP协议

    • 本质：是一个端到端的协议，越简单越好

    • 特点

    • 特点一：不需要建立长时间连接

    • 特点二：不断发送

    • 特点三：不进行流量控制

    • 特点四：Header文件小

    • 协议选择

    • 对时间不敏感的游戏，可以使用TCP

    • 对时间敏感的游戏，可以使用UDP

    • 但是大型的MMO，通常都是复合协议|两者混着使用

    • 不同的使用场合，使用不同协议完成

    • 补充

    • 游戏引擎是个工程科学，实用性至上

    • 可靠的UDP

    • 游戏服务器和其他服务器应用场合区别

    • 保证消息传递的Ordering顺序

    • 速度也得非常快

    • 而且可以实现群发

    • ARQ

    • 简介：自动回复的丢包检测|当发生丢包的时候，会告诉程序现在丢包了

    • 滑动窗口协议:滑动窗口中的数据依次进行发送，当收到了后续窗口中的数据时，即前面的数据已经成功接收

    • Stop And Wait ARQ：等待ACK后再发送下一个|速度很慢，效率很低

    • Go-Back-N ARQ：如果发现一个包丢了之后，只把这个窗口里面的数据重新发送一遍

    • 特点：十分使用

    • Selective Repeat ARQ：只通知丢了的包，因此连接更加的稳定

    • FEC

      • Forward Error Correction
      • 简介：收到了包，不管它，继续往前走，然后再重新找回被丢失的包|用额外的数据传输量来解决
      • 异或运算校验法：专门有值储存校验值。如果多个数据里面只丢掉一个，则可以轻松发现丢掉的数据并且恢复
      • Reed-Solomon Codes:冗余信息矩阵惩罚判断数据存放
      • 获得的信息乘以对应矩阵的逆就可以得到其原本结果

    • 时钟对齐

      • RTT
      • 简介：发出一个包之后，多久可以收回到包
      • 注意：其和Ping概念上十分相近，但是不同。Ping更加偏向于传输层

    • NTP

      • Network Time Protocol
      • GPS的实现：在天上有二十四颗卫星，每一颗都有极其准确的电子钟。你的GPS会接受它的信号，很精准的对其相位差，利用三角定位法算出其定位是多少，算出其位置
      • 定时的问题：任何内容的传输速度不能超过光速
      • Time Server Stratums：你和服务器来对时间

    • RPC

      • 问题：
        1. Socket 编程还有很多问题|当很多要回调的函数|数据要打包成数据包发送给别人，十分的复杂
        1. 客户端之间通信的时候，操作系统不同|IOS和Pai不同，且服务器通常在Linux上。而且数据有些是大端放置，有些是小端放置。数据的解码会有问题

    • 简介：

      • 让程序员只编写逻辑代码，剩下的怎么把消息打包这些，怎么进行处理都不管它。加密和解密都不用管，只要实现业务逻辑

    • 概念一：界面定义语言 | IDL：Interface Definition Language

    • 简介：所有网络通信的数据，都通过IDL定义好然后进行传输

    • RPC Stubs:每个RPC当中没有信息时，call了一个错误的RPC时，会有Stub Compiler来进行。

    • 概念二：RPC包的发送路径

    网络架构

    • P2P架构

    • 简介：早期游戏都是P2P类型，在局域网内进行

    • 特点：不防止作弊、简单

    • P2P With Host Server

    • 简介：有主机的P2P模式

    • 特点：开发商不需要花钱去维护服务器|现在很多沙盒游戏依然是这样（服务商提供服务器的检索服务，而不提供房间服务器）

    • 专用服务器

    • 简介：在服务器段维护统一的服务器，保证玩家统一稳定的链接

    • 补充：现代网络有很多的网络提供商家以及十分复杂的拓扑结构，因此很多游戏会自己接入服务商提供更好的网络服务|允许设置更告诉的端口

    游戏同步

    Game Synchronization

    核心矛盾：信息的实时性与信息的有效性之间的矛盾

    • 问题：

    • 1.网络多玩家的情况下，玩家的输入、逻辑该放在什么地方处理

    • 2.游戏之间平行世界二档同步

    • 快照同步 | Snapshot Synchronization

    • 简介：客户端只负责把玩家的输入发给服务器，服务器负责所有游戏逻辑的处理，服务器生成这一刻的游戏的快照，然后把快照发给所有的客户端玩家

    • 特点：

    • 1.结构干净 | 代码十分的清晰

    • 2.消耗十分多的算力 （现在客户端的算力越来越强，被浪费掉）

    • 3.服务器端和客户端的帧数可以不同

    • 4.服务器端的帧数更低

    • 5.对于带宽的要求很高

    • 快照压缩

    • 简介：很多快照里面数据在前后快照之间并不会发生变化，因此可以在打包Delta的时候进行优化

    • 用途：减少传输的数据量

    • 帧同步 | Lockstep Origin

    • 简介：所有的信息一致性的进行传输，然后一致性的进行处理

    • 实现：无论有多少客户端，将客户端的信息一致发给服务器，再统一的进行分发

    • 实现步骤一：Initialization

    • 概念：确保数据的一致高度相同，不允许任何的不同|偏差

    • 原因：因为游戏是复杂系统，只要有一点的偏差，偏差就会累计然后放大 | 因此需要高度一致

    • 实现步骤二：Deterministic

    • 概念：确认收到了所有人的输入之后，统一的将其进行分发，然后同时开始模拟下一帧的行为

    • 优点：概念清晰，公平

    • 缺点：当有客户端的信息拖拉，会导致其他客户端的进程被拖慢

    • 帧同步优化方案 | Bucket Synchronization

    • 概念：就算有客户端的数据没有被及时的上传上来也不理会，继续其他客户端的常规进程，放弃其未上传的数据（扔掉数据或者延迟处理）

    • 优点：保证网络的实时性

    • 缺点：可能导致部分玩家的输入的失效

    • 信息一致性 | Deterministic Difficulties

    • 问题

    • 1.数据容器|浮点数|算法|数学库|物理模拟|逻辑执行顺序|随机数 是否能保持一致|确定性，只有运算的一致性保持，才可以保证同步的完全一致

    • 2.现实世界并不完全一致|并不是牛顿式的完全一致性|不确定性原则当中的例子

    • 问题：数学函数与查表法

    • 很多复杂数学结果通过查表实现，但数学结构的表格可能不一致

    • 问题：定点数引擎

    • 在某些特定的游戏场合里面使用定点数引擎

    • 把其中最核心的部分业务放在定点数引擎中

    • 问题：随机数

    • 概念：分布式系统中随机数统一问题

    • 实现：客户端和服务器同步时，需要先把随机数的生成的种子进行同步

    • 实现：游戏引擎中的随机数种子在游戏生成的时候全部都会同步好

    • 帧同步中的调试问题

    • 原则：不停的将游戏的状态存储下来

    • 比如没过五帧使用checksum将游戏的状态信息记录

    • 网络延迟与抖动问题

    • 实现：各个节点生成大量的Buffer，把帧数据提前进行缓存

    • 问题一：服务器对客户端传输的帧波荡

    • 解决一：客户端有很多渲染帧，当网络出现问题时，画面不会因此出现抖动

    • 问题二：断线通连

    • 解决二：对游戏状态会有快照储存在磁盘中，在游戏恢复的时候，只要去追自己断线期间所经过的帧数。并且可以以更大的算了去处理断线帧的逻辑，将渲染的算力转移至此。

    • 延申：观战模式；对关键帧照快照，进而甚至可以实现回放

    • 帧同步问题

    • 问题一：作弊问题

    • 问题：客户端只是接收数据，但是可以在其中更改其数据与逻辑

    • 解决：服务器对比数据

    • 问题二：信息暴露问题

    • 问题：别人使用外挂获得更多场景的信息

    • 问题三：信息一致性难以同步

    • 问题四：断线后重连的追帧花费大

    • 优势一：可以实现对打击操作敏感的游戏，因为帧同步的实现的一致性

    • 优势二：方便实现游戏录屏

    状态同步

    大部分的MMORPG和社区类游戏使用状态同步

    核心：将用户的状态信息储存进去，包括部分关键性信息

    服务器端会在其上面维护一个完整的全信息宇宙，安全更高

    • 权威服务器与命令

    • 服务器：服务器说你不对，你就不对。同时客户端中有一个决定者

    • 动作实例

    • 一：客户端发送命令

    • 二：服务器接受命令

    • 概念1：服务器计算命令结果，把计算的结果发送到全部的客户端

    • 概念2：每个人提出自己的动作，服务器判定动作是否有效，最后再把动作发送下去

    • 三：服务器将命令发送给所有客户端

    • Client Side Prediction

    • 例子：守望先锋

    • 概念：客户端永远先于服务器半个RTT的时间，并且具有一个缓冲帧，并利用本地的预测（进行平滑的插值算法）

    • 概念：Server Reconciliation

    • 不只是存当前的状态，还要把之前一点的状态也储存下来，然后进行比对，如果比对核实则表示其正确
